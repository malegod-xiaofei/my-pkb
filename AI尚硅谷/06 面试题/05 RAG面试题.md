# 05 RAG面试题

> 参考笔记：[[01-RAG入门]]、[[02-文本向量化]]、[[03-向量数据库存储与检索]]、[[04-从零到一手动搭建RAG系统]]、[[05-RAG优化]]、[[06-Weaviate向量数据库]]、[[07-LangChain使用之Retrieval]]。

## 1. 什么是 RAG？

**答案要点：**

RAG 是 Retrieval-Augmented Generation，检索增强生成。它在大模型生成回答前，先从外部知识库中检索与问题相关的内容，再把检索结果作为上下文交给大模型生成答案。

核心价值：

- 缓解大模型幻觉。
- 补充私有知识和实时知识。
- 降低直接微调模型的成本。
- 让答案具备可追溯来源。

## 2. 为什么需要 RAG？

**答案要点：**

大模型存在几个典型问题：

- 幻觉：可能编造看似合理但错误的答案。
- 知识过时：训练数据有截止时间，无法天然知道最新信息。
- 私有知识缺失：模型不知道企业内部文档、制度、数据库内容。
- 上下文窗口有限：无法无限输入全部资料。

RAG 通过“检索外部知识 + 基于上下文生成”来缓解这些问题。

## 3. RAG 的基本流程是什么？

**答案要点：**

离线阶段：

1. 文档收集。
2. 文档清洗和解析。
3. 文本切分。
4. Embedding 向量化。
5. 存入向量数据库。

在线阶段：

1. 用户提出问题。
2. 问题向量化。
3. 相似度检索 Top-K 文档片段。
4. 可选 Rerank 重排。
5. 构造 Prompt。
6. 大模型基于上下文生成答案。
7. 返回答案和引用来源。

## 4. 文本切分为什么重要？

**答案要点：**

文本切分直接影响召回质量。

- Chunk 太大：包含噪声多，匹配不精准，也占用上下文。
- Chunk 太小：上下文不完整，模型无法理解完整语义。
- Overlap 太小：跨段语义可能断裂。
- Overlap 太大：重复内容多，增加成本和噪声。

常用策略：按标题切分、按段落切分、递归字符切分、语义切分、滑动窗口切分。

## 5. Embedding 模型的作用是什么？

**答案要点：**

Embedding 模型将文本转换为稠密向量，使语义相近的文本在向量空间中距离更近。

在 RAG 中的作用：

- 文档向量化。
- 用户问题向量化。
- 通过相似度检索找到相关片段。

选择 Embedding 模型时要考虑语言、领域、维度、性能、成本和与向量数据库的适配。

## 6. 向量数据库的作用是什么？常见向量数据库有哪些？

**答案要点：**

向量数据库用于存储文档向量，并支持高效相似度检索。

常见产品：

- FAISS：本地向量检索库，轻量高效。
- Milvus：开源分布式向量数据库。
- Weaviate：支持向量检索和对象存储。
- Chroma：适合本地和原型开发。
- Qdrant：Rust 实现，性能和过滤能力较好。
- Pinecone：云端托管向量数据库。
- Elasticsearch / OpenSearch：支持 BM25 和向量混合检索。
- PostgreSQL + pgvector：适合已有 PostgreSQL 技术栈。

## 7. 常见相似度计算方式有哪些？

**答案要点：**

- Cosine Similarity：关注方向相似度，常用于文本向量。
- Dot Product：点积，常用于归一化向量或特定训练目标的 embedding。
- Euclidean Distance：欧氏距离，衡量空间距离。

具体选择要和 Embedding 模型训练方式、向量是否归一化以及数据库索引方式保持一致。

## 8. Top-K 应该如何选择？

**答案要点：**

Top-K 表示检索返回相似度最高的 K 个片段。

- K 太小：可能漏掉关键资料，召回不足。
- K 太大：引入噪声，占用上下文，增加成本。

选择方法：

- 根据问题类型、chunk 大小、上下文窗口调参。
- 结合 Rerank 筛选最终片段。
- 通过评测集比较不同 K 值下的答案质量。

## 9. 什么是 Rerank？为什么需要 Rerank？

**答案要点：**

Rerank 是对初步召回结果进行二次排序的过程。

向量检索通常速度快、召回广，但排序未必最精准。Rerank 模型通常会同时看 query 和文档内容，判断相关性更准确。

作用：

- 提升最终上下文相关性。
- 降低噪声片段进入 Prompt 的概率。
- 改善答案准确性和忠实度。

常见流程：先向量检索召回较多候选，再用 Rerank 选出少量高质量片段。

## 10. 什么是混合检索？

**答案要点：**

混合检索是将关键词检索和向量检索结合起来。

- 关键词检索：如 BM25，适合精确词、编号、专有名词、错误码。
- 向量检索：适合语义相似、同义表达、自然语言问题。

混合检索能兼顾精确匹配和语义召回，常用于企业知识库问答。

## 11. RAG 为什么仍然可能产生幻觉？

**答案要点：**

RAG 只能降低幻觉，不能完全消除幻觉。

可能原因：

- 检索结果不相关。
- 关键资料没有被召回。
- 文档本身错误或过时。
- Prompt 没有限制模型必须基于上下文回答。
- 上下文太长，模型忽略关键内容。
- 模型对检索内容进行了不当推理或补充。

解决方向：优化文档质量、切分、检索、重排、Prompt、引用和评测。

## 12. 如何优化 RAG 效果？

**答案要点：**

可以从全链路优化：

- 数据侧：清洗文档、去重、保留标题结构、处理表格和图片 OCR。
- 切分侧：根据文档结构选择 chunk size 和 overlap。
- 检索侧：选择合适 Embedding、向量库、Top-K、元数据过滤。
- 排序侧：增加 Rerank。
- 生成侧：优化 Prompt，要求基于资料回答、不知道就说明无法确定。
- 评测侧：构建问答评测集，关注召回率、答案正确性、忠实度、引用准确性。

## 13. 什么是 GraphRAG？适合什么场景？

**答案要点：**

GraphRAG 是在传统 RAG 基础上引入知识图谱或图结构的增强方法。

它通过抽取实体、关系和社区摘要，把文档组织成图结构，再结合图检索和向量检索回答问题。

适合场景：

- 多跳推理。
- 复杂关系查询。
- 跨文档综合分析。
- 需要可解释关系路径的问答。

代价是构建流程更复杂，维护成本更高。

## 14. 什么是 Agentic RAG？

**答案要点：**

Agentic RAG 是把 Agent 能力引入 RAG 流程，让模型不只是执行一次检索，而是可以根据任务动态规划和多轮检索。

特点：

- 能拆解复杂问题。
- 能决定检索哪些知识库或工具。
- 能根据中间结果继续检索。
- 能结合工具调用、计算、数据库查询生成最终答案。

适合复杂问答、跨系统查询和多步骤任务。

## 15. 如何评测 RAG 系统？

**答案要点：**

评测可以分为检索和生成两部分。

检索指标：

- Recall@K：正确资料是否在前 K 个结果中。
- Precision@K：前 K 个结果中有多少相关。
- MRR / NDCG：排序质量。

生成指标：

- 答案正确性。
- 上下文忠实度。
- 引用准确性。
- 幻觉率。
- 可读性。

工程指标：延迟、成本、成功率、用户满意度。

## 16. 企业 RAG 落地要注意哪些问题？

**答案要点：**

- 权限控制：不同用户只能检索有权限的文档。
- 数据更新：知识库要支持增量更新和版本管理。
- 文档质量：格式混乱、扫描件、表格、图片会影响解析。
- 安全合规：敏感信息脱敏，日志审计。
- 可观测性：记录检索片段、Prompt、模型回答和用户反馈。
- 成本控制：控制 chunk、Top-K、上下文长度和模型选择。
