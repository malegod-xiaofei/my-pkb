# 08 本地部署、评测与安全面试题

> 参考笔记：[[Ollama使用]]、[[01-企业级大模型的部署]]、[[02-大模型实战项目：Agent&RAG]]、[[01-RAG入门]]、[[05-RAG优化]]、[[c-01-MCP入门]]。

## 1. Ollama 是什么？

**答案要点：**

Ollama 是一个便于在本地运行和管理大语言模型的工具，支持拉取模型、运行模型、创建自定义模型配置，并提供本地 API 调用能力。

常见用途：

- 本地聊天和学习。
- 本地 RAG 原型验证。
- 离线或内网场景。
- 隐私要求较高的实验环境。
- 与 Spring AI、LangChain 等框架集成。

## 2. 本地模型和云端模型有什么区别？

**答案要点：**

本地模型：

- 优点：数据更可控，隐私性好，无按量 API 调用费，可离线。
- 缺点：硬件要求高，部署维护复杂，模型能力可能弱于顶级闭源模型。

云端模型：

- 优点：能力强，部署简单，弹性好，升级方便。
- 缺点：有调用成本，依赖网络，存在数据合规和隐私问题。

企业常根据数据敏感程度、成本、效果和运维能力做混合选型。

## 3. 本地部署大模型需要关注哪些资源？

**答案要点：**

- GPU 显存：决定能运行多大的模型和上下文。
- CPU 和内存：影响加载、预处理和 CPU 推理。
- 磁盘：存储模型权重和向量库。
- 网络：模型下载和服务访问。
- 并发能力：影响服务吞吐。
- 上下文长度：影响显存和延迟。

模型参数越大、上下文越长、并发越高，资源压力越大。

## 4. 什么是模型量化？

**答案要点：**

量化是将模型权重或激活从高精度转换为低精度，例如 FP16 转 INT8、INT4。

优点：

- 降低显存和内存占用。
- 提升推理速度。
- 降低部署成本。

缺点：

- 可能损失精度。
- 数学、代码、复杂推理能力可能受影响。
- 不同模型和量化方法效果不同，需要评测。

## 5. 如何选择本地部署模型？

**答案要点：**

选择维度：

- 任务类型：通用问答、代码、中文、Embedding、多模态、Rerank。
- 硬件资源：显存、内存、CPU/GPU。
- 模型规模：小模型速度快，大模型效果更好但成本高。
- 量化版本：平衡效果和资源。
- 上下文长度：是否支持长文档。
- 许可证：是否允许商用。
- 评测结果：用业务数据实际测试。

## 6. vLLM 的特点是什么？

**答案要点：**

vLLM 是高吞吐的大模型推理服务框架，核心特点是 PagedAttention，用于更高效地管理 KV Cache。

优势：

- 支持高并发推理。
- 提升吞吐。
- 适合服务化部署。
- 通常支持 OpenAI API 兼容接口，便于应用接入。

## 7. 什么是 KV Cache？为什么重要？

**答案要点：**

在自回归生成中，模型每生成一个 token 都需要关注之前的 token。KV Cache 会缓存历史 token 的 Key 和 Value，避免重复计算。

作用：

- 加速解码阶段。
- 降低重复计算。

但 KV Cache 会占用显存，且上下文越长、并发越高，占用越大。

## 8. 大模型评测可以分为哪些类型？

**答案要点：**

- 通用能力评测：语言理解、知识、推理、数学、代码。
- 领域评测：医疗、法律、金融、企业知识。
- RAG 评测：检索召回、答案忠实度、引用准确性。
- Agent 评测：任务完成率、工具调用准确率、安全性。
- 安全评测：越狱、隐私泄露、有害内容。
- 在线评测：A/B 测试、用户满意度、点击和反馈。

## 9. 如何构建企业内部大模型评测集？

**答案要点：**

1. 收集真实业务问题。
2. 覆盖高频问题、边界问题、异常问题和安全问题。
3. 标注标准答案或评分规则。
4. 区分开发集和测试集，避免过拟合评测集。
5. 定期更新，反映新业务和新文档。
6. 保留问题、检索结果、模型回答和人工评分记录。

## 10. LLM-as-a-Judge 是什么？有什么问题？

**答案要点：**

LLM-as-a-Judge 是用大模型作为评审器，对另一个模型的输出进行打分或比较。

优点：

- 成本低。
- 速度快。
- 适合大规模初筛。

问题：

- 评审模型可能有偏见。
- 对提示词敏感。
- 可能偏好更长或更流畅的答案。
- 对专业领域可能判断不准。

因此关键评测仍建议结合人工评审。

## 11. RAG 评测重点关注什么？

**答案要点：**

检索侧：

- Recall@K。
- Precision@K。
- MRR / NDCG。
- 关键证据是否被召回。

生成侧：

- 答案正确性。
- 上下文忠实度。
- 引用准确性。
- 幻觉率。
- 可读性。

工程侧：延迟、成本、失败率、用户反馈。

## 12. Agent 评测重点关注什么？

**答案要点：**

- 任务完成率。
- 工具选择准确率。
- 工具参数正确率。
- 执行步骤合理性。
- 失败恢复能力。
- 是否越权调用工具。
- 成本和延迟。
- 是否出现无限循环或无效循环。

Agent 评测要关注“过程”和“结果”，不能只看最终回答是否流畅。

## 13. 什么是 Prompt Injection？

**答案要点：**

Prompt Injection 是指用户输入或外部文档中包含恶意指令，诱导模型忽略系统规则、泄露信息或错误调用工具。

典型场景：

- 用户直接输入“忽略之前所有指令”。
- RAG 文档中夹带恶意提示。
- 网页内容诱导 Agent 执行危险操作。

防护重点是区分“指令”和“数据”，并限制工具权限。

## 14. 如何防范 Prompt Injection？

**答案要点：**

- 系统提示词明确外部内容只作为数据，不作为指令。
- 对 RAG 文档做清洗和风险检测。
- 工具调用前做权限和参数校验。
- 高风险操作要求人工确认。
- 限制模型可访问的敏感信息。
- 对工具返回内容进行隔离和标记。
- 建立攻击样例评测集。

## 15. 大模型应用有哪些常见安全风险？

**答案要点：**

- 幻觉导致错误决策。
- 隐私和敏感数据泄露。
- Prompt Injection。
- 越狱攻击。
- 工具越权调用。
- RAG 检索到无权限文档。
- 数据投毒。
- 有害内容生成。
- 版权和合规风险。

## 16. RAG 中如何做权限控制？

**答案要点：**

- 文档入库时保存权限元数据，例如部门、角色、用户、密级。
- 检索时根据当前用户权限过滤候选文档。
- 不允许先检索后再让模型判断权限。
- 答案引用也要遵守权限。
- 日志中避免记录用户无权访问的敏感内容。

一句话：权限过滤必须发生在检索和上下文拼接之前。

## 17. Tool Calling 和 MCP 工具调用如何做安全控制？

**答案要点：**

- 工具白名单。
- 最小权限。
- 参数 schema 校验。
- 业务鉴权。
- 高风险操作二次确认。
- 超时和最大调用次数限制。
- 沙箱隔离。
- 执行日志和审计。
- 禁止把密钥、令牌直接暴露给模型。

## 18. 大模型应用如何控制成本？

**答案要点：**

- 选择合适模型：简单任务用小模型，复杂任务用强模型。
- 控制上下文长度和输出长度。
- RAG 检索结果压缩和重排。
- 对重复问题做缓存。
- 使用流式输出提升体验但仍要限制最大 token。
- 监控调用量、失败率和重试率。
- 本地模型和云端模型混合使用。

## 19. 大模型应用上线前需要检查哪些内容？

**答案要点：**

- 效果评测是否达标。
- 失败案例是否有处理策略。
- Prompt 和知识库是否版本化。
- 是否有日志、监控和告警。
- 是否设置超时、重试、限流、熔断。
- 是否完成权限控制和敏感数据脱敏。
- 是否有成本预算和用量监控。
- 是否支持降级和人工兜底。
