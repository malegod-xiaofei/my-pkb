## 一，标准工业环境的划分
### 学习目标
+ 了解标准工业环境的划分以及作用。
+ 从环境角度探索算法工程化的重要点。

---

#### 标准工业环境
+ 研发环境
+ 测试环境
+ 生产环境

---

+ 研发环境作用：
    - 算法工程师完成模型训练的流水线所需的环境，包括持久化训练数据，数据分析，特征处理，模型训练，模型验证以及调优，模型保存和上传等。
+ 研发环境的配置：
    - 因为模型训练在此环境下发生，因此它是所需算力最大的环境，往往需要配有GPU来支持训练。 

---

+ 测试环境作用：
    - 用于测试模型加载服务的关键指标，保证模型的正常预测功能以及相关工程化措施。
+ 测试环境的配置：
    - 只发生模型的预测过程，因此一般情况下，算力要求不高。正常4C，8G即可满足要求。

---

+ 生产（正式）环境作用:
    - 测试的服务满足上线指标要求后，即可将代码和模型环境部署在生产环境中。供线上请求调用。
+ 生产（正式）环境配置:
    - 模型预测环境与测试环境相同，但是需要进行多节点分布，来保证高可用。

---

+ 接下来我们将对三个环境进行具体的解析，来理解整个算法工程化的过程。其中重点包括：
    - 版本控制与S3
    - 服务日志
    - 模型风险控制
    - 服务在线指标
    - Git提交
    - 线上部署

---

### 小节总结
+ 标准工业环境划分为：
    - 研发环境
    - 测试环境

+ 您通过 S3 存储和检索的资产被称为对象。对象存储在存储段（bucket）中。您可以用硬盘进行类比：对象就像是文件，存储段就像是文件夹（或目录）。与硬盘一样，对象和存储段也可以通过统一资源标识符（Uniform Resource Identifier，URI）查找。

---

#### 什么是服务日志及其作用
+ 是一个或多个由服务器自动创建和维护的文件，其中包含其所执行活动的内容。
+ 服务日志在工业生产中非常重要，负责服务的信息打印，比如异常查询，数据指标的统计等等。

---

#### 日志级别
+ 打印的内容一般会按照功能分为不同的级别（算法服务一般分为四个级别，后端工程一般有8个级别，当然日志级别是可以自定义的）：
    - DEBUG：测试过程中，寻找bug时打印的信息，不会在生产环境中出现。 
    - INFO：正常的输出信息，作为某些功能正常启用的提示。
    - WARNING：警告信息，作为某些功能可能出现异常的警告。
    - ERROR：错误信息，说明某些代码运行出现了错误，需要及时修正。

---

#### 日志切割
+ 默认情况下，日志会被写入到指定的文件中，但是一旦在某天服务出现问题，我们很难做定位，因此，需要将日志按天分割，当然，这也方便之后的日志存档。

---

#### 日志存档
+ 随着时间推移，日志会越来越多，但是机器的磁盘空间是有限的，因此，需要在服务器上只保留3天的日志信息，定期将多余日志存档转移到专门的硬盘存储。

---

#### 日志实现工具：logging
+ 我们知道supervisor中可以帮助我们打印日志，并控制日志文件的大小，但对于日志分割却无能为力，因此我们需要使用python中专门的日志处理工具包：logging，下面让我们来学习使用它。logging的功能有很多，我们这里只去介绍工程化中用到的方法。

---

+ 按时间切割的实现：

```plain
import os
# 从logging的处理器中导入时间切片处理器
from logging.handlers import TimedRotatingFileHandler

# 定义日志的写入路径
log_root = "./"
log_path = log_root + "flask.log"
if not os.path.exists(log_root):
    os.mkdir(log_root)


# 日志处理器主要配置日志处理方式的相关参数
# filename: 日志存储路径+日志名称
# when: 按什么时间分割，D: 天，H：小时，M：分钟，S：秒
# interval: 时间的数量，比如3天或15分钟
# backupCount: 后台日志的数量，保存多少份时停止，防止磁盘溢出
# encoding: 日志内容编码
# delay: 是否立即生效
# utc: 是否按照原子钟世界标准时间，否则使用本地时间
handler = TimedRotatingFileHandler(
    filename=log_path,
    when="D",
    interval=1,
    backupCount=15,
    encoding="UTF-8",
    delay=False,
    utc=True,
)

# 过往日志后缀
# eg: flask.log.2020-12-08 
handler.suffix = "%Y-%m-%d"
```

+ 规范的日志样式实现：

```plain
import logging

# 定义日志输出格式
LOG_FORMAT = logging.Formatter("%(asctime)s - %(levelname)s - %(message)s")
# 设置处理器格式
handler.setFormatter(LOG_FORMAT)

# 实例化一个日志对象，日志对象具有添加某个处理器的功能
logger = logging.getLogger(__name__)
# 添加处理器
logger.addHandler(handler)
# 设置日志打印级别，默认不打印低于INFO（包括）的日志
logger.setLevel(logging.INFO)
# 打印具体信息
# 实际生产中，将打印模型的输入输出等关键信息
logger.info('info message')
```

## 五，A／B测试
### 学习目标
+ 了解什么是A/B测试。
+ 了解什么是分流。
+ 掌握算法分流的逻辑与实现。

---

#### 什么A/B测试
+ AB测试是为Web或App界面或流程制作两个（A/B）或多个（A/B/n）版本，在同一时间维度，分别让组成成分相同（相似）的访客群组（目标人群）随机的访问这些版本，收集各群组的用户体验数据和业务数据，最后分析、评估出最好版本，正式采用。
+ 在算法工程化中主要为了验证新方法/新模型的上线效果。

---

#### 什么是分流
+ 具体来讲，最简单A/B testing一般包括两种对比逻辑：
    - 从服务端分流的小流量整体对比实验：90%流量按照原有逻辑，10%的流量分给新的模型预测。
    - 从模型服务进行分流对比：因为小流量对比实验受到一些极端情况的影响，比如线上分类不均衡。模型在小流量上给出了接近统一的结果，为了监控和避免这种情况，我们会在服务中再将10%的中模型预测成1的数据进行分流，成为实验组和对照组，这样我们就是真正在比较模型预测成1的价值。90%的流量称为旧算法组。

---

#### 分流的逻辑与实现
+ 按照userId进行映射。
+ 分流的实现：

```plain
result = {}
# 模型的预测值大于某个阈值
if y_pred > THREDHOLD:
    # 对模型判断大于阈值的用户进行分桶
    # 以验证模型的判断（以及后面的对应的策略）
    # 施加给该用户群体是有效的
    if int(uid) % 2 == 1:
        # 风险监控器
        # res = control_counter(uid, result)
        # if res: return res
        result.update({"vipLevel": 1})
    else:
        result.update({"vipLevel": 0})
else:
    result.update({"vipLevel": 0})
```

## 六，模型服务风险监控
### 学习目标
+ 了解什么是模型服务风险监控。
+ 掌握模型服务风险监控的实现。

#### 模型服务风险监控
+ 模型服务风险监控是指针对模型服务在线上可能给业务带来的风险，进行一系列控制的措施。 算法服务一般“镶嵌”在业务服务里面，所以模型服务的风险监控一般是比较薄弱的，但是有一些非常重要的监控需要添加来防止灾难。

#### 模型服务风险监控过程
+ 首先需要最坏情况评估：是否占用线上其他优质资源。
    - 例子：新用户推荐，为了提升新用户的转化，强烈推荐优质资源，导致vip用户付费下降。
+ 如何控制该风险？
    - 保证每小时不同用户请求的总次数。
+ 风险控制的实现：

```plain
# 每小时限制不同uid的请求总次数
# limit_counter存储不同的uid, 使用有序集合数据结构
# 每个小时进行清空
limit_counter = OrderedSet()

# 服务启动的开始时间
start_time = int(time.time())
# 从服务启动开始，过去的小时数
hour_num = 0

# 每小时最多响应的用户数
USER_PER_HOUR = 3000


def control_counter(uid, result):
    """整点判断, 并将uid列表清空"""
    # 将limit_counter置为全局变量
    global limit_counter
    # 向集合中添加uid（1小时内添加到一定数量后，之后的uid将不会被推荐）
    limit_counter.add(uid)
    # 当前的“小时数”
    now_hour_num = (int(time.time()) - start_time) // 3600
    # 判断当前“小时数”是否大于之前的小时数（是否过了一个小时）
    global hour_num
    if now_hour_num > hour_num:
        hour_num = now_hour_num
        # 清空集合
        limit_counter = OrderedSet()
    else:
        # 判断集合总长度是否大于USER_PER_HOUR
        if len(limit_counter) > (USER_PER_HOUR):
            # 注意：这里是有序集合的作用，业务上要保证一旦给某个用户推荐了属于优质的视频，那么优质视频推荐应该是持续的。
            if uid not in limit_counter[:USER_PER_HOUR]:
                result.update({"vipLevel": 0})
                res = {"code": 600, "data": {}}
                res["data"] = result
                return res
            else:
                pass
    return
```

### 学习目标

---

#### 在线服务的重要指标
+ RT(requests time):
    - 整个模型服务响应时间，服务中模型预测的总体时间。
+ RT要求:
    - 一般情况下，对于算法服务的RT要求小于100ms。
+ Flask框架下常规模型的响应时间：
    - sklearn实现的LR：<10ms
    - sklearn实现的GBDT：<20ms
    - Pytorch实现的MLP：<70ms
    - Tensorflow实现的MLP：<90ms
    - Pytorch实现的BERT：<100ms
    - Tensorflow实现的BERT：<100ms

---

---

+ QPS：
    - 每秒处理请求次数，与APP日活和峰值时刻有关
+ QPS一般对应量级:
    - 日活30万左右，服务QPS>20
    - 日活100万左右，服务QPS>80
    - 日活1000万左右，服务QPS>500
    - 日活3亿左右，服务QPS>5000

---

#### 在线服务的重要指标的优化方式
+ 离线模式：
    - eg：如用户推荐序列，采集用户的离线特征，模型将所有用户进行预测并将结果写入缓存数据库（redis），当某个用户请求在线上触发时，直接取redis数据。
+ 离线模式的缺点：
    - 模型预测往往是T+1的模式，很多新用户信息特征无法有效的进行利用。eg：新用户推荐时（冷启动），无法使用新用户信息，如手机型号，国家地区等。
+ 解决方法：
    - 算法服务工程化。

#### 正式环境的节点申请
+ 考虑业务端服务器的位置，因为算法服务是内嵌服务，所以需要在同一个region。
+ 为了保证RT和QPS的重要指标，至少开启四个节点（线上集群），机器情况一般与测试机相同（4C，8G）。

#### Docker部署
+ Docker 是一个开源的应用容器引擎，让开发者可以打包他们的应用以及依赖包到一个可移植的镜像中，然后发布到任何流行的 Linux或Windows 机器上，也可以实现虚拟化。容器是完全使用沙箱机制，相互之间不会有任何接口。
+ 将运行环境打包成虚拟容器，使用DockerFile进行服务启动命名的编写。

#### K8S
+ k8s全称kubernetes，k8s是为容器服务而生的一个可移植容器的编排管理工具, 理解为对Docker的管理系统，比如Docker的资源分配，不同的虚拟化产品的融合。 

#### 滚动重启
+ 保证每次更新上线每个节点重启是滚动的，这样服务可以保持不中断。

#### 大数据落表
+ Hive
    - hive是基于Hadoop的一个数据仓库工具，用来进行数据提取、转化、加载，这是一种可以存储、查询和分析存储在Hadoop中的大规模数据的机制。hive数据仓库工具能将结构化的数据文件映射为一张数据库表，并提供SQL查询功能，能将SQL语句转变成MapReduce任务来执行。简单理解为分布式的mysql。
+ 使用python连接Hive

```plain
from pyhive import hive

HIVE_CONFIG = {
       "host": 0.0.0.0,
       "post": 10001,
       "username": "admin"
       "database": "default"}

conn = hive.Connection(**HIVE_CONFIG)
cursor = conn.cursor()
cursor.execute('select * from testhive limit 10')
for result in cursor.fetchall():
    print(result)
```
