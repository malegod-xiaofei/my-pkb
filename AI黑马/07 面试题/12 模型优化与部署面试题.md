# 12 模型优化与部署面试题

> 参考笔记：`AI黑马/传智项目文档/11 模型优化`、`AI黑马/传智项目文档/07 传智大脑/7. 模型压缩与推断加速技术`、`AI黑马/06 文本摘要(HTML)/7. 模型的部署与总结`。
>
> 权威参考：TensorFlow Serving、TensorFlow Lite、Hugging Face Transformers、ONNX Runtime、PyTorch Quantization 相关官方文档。

## 1. 模型部署通常包括哪些步骤？

**答案要点：**

1. 模型训练和评估。
2. 导出模型文件和配置。
3. 编写预处理和后处理逻辑。
4. 封装推理服务。
5. 压测延迟和吞吐。
6. 部署到服务器或容器环境。
7. 接入监控、日志和报警。
8. 灰度发布和回滚。

## 2. 离线训练和在线推理有什么区别？

**答案要点：**

- 离线训练关注模型效果、训练稳定性和实验复现。
- 在线推理关注低延迟、高吞吐、稳定性和资源消耗。
- 离线可使用复杂特征和大批量数据，在线需要保证特征实时可得。
- 线上线下预处理必须一致，否则会导致效果下降。

## 3. 模型量化是什么？

**答案要点：**

模型量化是用低精度数值表示模型权重和激活，例如从 FP32 转为 INT8。

作用：

- 减小模型体积。
- 降低内存和显存占用。
- 提升推理速度。
- 便于边缘设备部署。

可能代价是模型精度下降，需要校准或量化感知训练。

## 4. 动态量化和静态量化有什么区别？

**答案要点：**

- 动态量化通常只量化权重，激活在推理时动态计算缩放参数，实现简单。
- 静态量化会提前用校准数据统计激活范围，同时量化权重和激活，推理速度提升更明显。
- 静态量化部署前需要代表性校准数据。
- 量化感知训练在训练过程中模拟量化误差，通常精度保持更好。

## 5. 知识蒸馏是什么？

**答案要点：**

知识蒸馏用大模型作为教师模型，引导小模型学习。

- 教师模型输出软标签，包含类别之间的相似性信息。
- 学生模型结构更小，推理更快。
- 蒸馏常用于模型压缩和移动端部署。
- 可以结合真实标签损失和蒸馏损失训练。

## 6. 模型剪枝是什么？

**答案要点：**

模型剪枝通过删除不重要的权重、通道、神经元或层来减少计算量和模型大小。

类型：

- 非结构化剪枝：删除单个权重，稀疏度高但硬件加速不一定明显。
- 结构化剪枝：删除通道、卷积核或层，更利于实际加速。

剪枝后通常需要微调恢复精度。

## 7. 模型推理慢可能有哪些原因？

**答案要点：**

- 模型参数量和计算量大。
- 输入序列或图像尺寸过大。
- batch 设置不合理。
- 预处理或后处理成为瓶颈。
- CPU/GPU 数据拷贝频繁。
- 没有使用推理加速框架。
- 并发设计不合理。

## 8. 如何优化模型推理速度？

**答案要点：**

- 使用更小模型或蒸馏模型。
- 量化、剪枝、低精度推理。
- 使用 ONNX Runtime、TensorRT、TensorFlow Serving 等推理框架。
- batch 推理。
- 缓存高频请求结果。
- 优化预处理和后处理。
- 减少输入长度或图像分辨率。
- 异步队列和并发服务。

## 9. CPU 部署和 GPU 部署有什么区别？

**答案要点：**

- CPU 部署成本低、通用性强，适合小模型和低并发场景。
- GPU 并行计算能力强，适合大模型、高吞吐和批量推理。
- GPU 对 batch 更敏感，batch 太小可能利用率不高。
- CPU 需要关注线程数和向量化优化，GPU 需要关注显存和数据传输。

## 10. 什么是 batch 推理？

**答案要点：**

batch 推理把多个请求合并成一个批次输入模型，提升硬件利用率和吞吐。

优点：

- 提升 GPU 利用率。
- 降低单位样本推理成本。

缺点：

- 可能增加单个请求等待时间。
- 需要在延迟和吞吐之间平衡。

## 11. 模型服务如何做并发控制？

**答案要点：**

- 使用 Web 服务框架处理请求。
- 使用队列做削峰。
- 控制 worker 数和线程数。
- 对 GPU 推理做批处理。
- 设置超时和限流。
- 对高成本请求做输入长度限制。
- 监控 QPS、延迟、错误率和资源使用率。

## 12. 什么是模型版本管理？

**答案要点：**

模型版本管理记录模型文件、代码、数据、配置和指标之间的对应关系。

作用：

- 支持实验复现。
- 支持线上回滚。
- 便于对比不同模型效果。
- 追踪问题来源。

通常需要记录训练数据版本、特征版本、代码 commit、超参数和评估结果。

## 13. 线上模型需要监控哪些指标？

**答案要点：**

系统指标：

- QPS。
- 平均延迟和 P95/P99 延迟。
- CPU、内存、GPU、显存。
- 错误率和超时率。

模型指标：

- 预测分布。
- 置信度分布。
- 数据漂移。
- 业务转化指标。
- 人工抽检准确率。

## 14. 什么是数据漂移和概念漂移？

**答案要点：**

- 数据漂移：线上输入数据分布相对训练数据发生变化。
- 概念漂移：输入和标签之间的关系发生变化。

应对方式：

- 持续监控特征和预测分布。
- 定期采样标注。
- 增量训练或周期性重训。
- 保留回滚机制。

## 15. 如何做灰度发布？

**答案要点：**

灰度发布是先让一小部分流量使用新模型，观察效果和稳定性，再逐步扩大流量。

关注：

- 流量切分策略。
- 新旧模型指标对比。
- 异常自动报警。
- 快速回滚。
- A/B 测试统计显著性。

## 16. 如何保证训练和推理预处理一致？

**答案要点：**

- 把预处理逻辑封装为统一模块。
- 使用 Pipeline 管理特征处理和模型。
- 保存 tokenizer、词表、标准化参数等工件。
- 上线前做端到端样本对齐测试。
- 避免训练和线上各写一套逻辑。

## 17. BERT 类模型部署时如何优化？

**答案要点：**

- 限制最大序列长度。
- 使用蒸馏模型，如 DistilBERT。
- 使用量化。
- 使用 ONNX Runtime 或 TensorRT。
- batch 推理。
- 缓存 tokenizer 和高频请求结果。
- 对长文本先切分或召回相关片段。

## 18. 文本生成模型部署有哪些特殊问题？

**答案要点：**

- 生成是逐 token 解码，延迟较高。
- 输出长度不确定，需要限制最大长度。
- 需要处理重复生成和低质量输出。
- 需要内容安全和敏感词过滤。
- 需要流式输出提升体验。
- 需要监控 token 级成本和超时。

## 19. 如果模型上线后误判严重，如何排查？

**答案要点：**

排查路径：

1. 检查线上输入和训练输入格式是否一致。
2. 检查 tokenizer、词表、标准化参数是否正确加载。
3. 检查模型版本是否部署正确。
4. 抽样 bad case，判断是否数据分布变化。
5. 检查阈值和后处理规则。
6. 对比离线复现结果和线上日志。
7. 必要时回滚到稳定版本。

## 20. 面试中如何描述一次模型优化经历？

**答案要点：**

建议结构：

1. 原始问题：效果差、延迟高、显存占用大或线上不稳定。
2. 定位过程：指标分析、bad case、性能 profiling。
3. 优化方案：数据、模型、训练、推理或系统优化。
4. 实验对比：优化前后指标变化。
5. 上线结果：业务指标、资源节省、稳定性提升。

重点是说明“为什么这样优化”和“优化带来了什么可验证收益”。
