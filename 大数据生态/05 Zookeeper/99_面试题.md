# ZooKeeper 高频面试题

## 整理依据

基于本目录 ZooKeeper 知识点、集群安装、配置参数、ZNode、监听器、客户端 API、动态上下线、分布式锁、ZAB 协议和源码解析等笔记整理，重点覆盖分布式协调、选举、一致性、生产部署和常用命令。

## 一、基础概念与应用场景

### 1. ZooKeeper 是什么？

ZooKeeper 是一个开源的分布式协调服务，为分布式系统提供统一命名、配置管理、集群管理、节点动态上下线感知、分布式锁、软负载均衡等能力。

### 2. ZooKeeper 的数据模型是什么？

ZooKeeper 的数据模型类似 Unix 文件系统，是一棵树。每个节点称为 ZNode，通过路径唯一标识，每个 ZNode 默认最多可存储约 1MB 数据。

### 3. ZooKeeper 适合存储大数据吗？

不适合。ZooKeeper 适合存储少量协调元数据，如配置、状态、锁节点、服务注册信息等。单个 ZNode 默认限制约 1MB，不适合作为业务大数据存储系统。

### 4. ZooKeeper 常见应用场景有哪些？

常见场景包括统一命名服务、统一配置管理、统一集群管理、服务器动态上下线监听、分布式锁、Master 选举、服务注册发现和软负载均衡。

## 二、集群部署与配置

### 5. ZooKeeper 集群为什么通常部署奇数台？

ZooKeeper 依赖过半机制，奇数台能在相同容错能力下减少机器浪费。例如 3 台和 4 台都只能容忍 1 台故障，但 4 台成本更高。

### 6. 生产集群部署多少台 ZooKeeper 合适？

课程经验中，10 台服务器可部署 3 台 ZooKeeper，20 台可部署 5 台，100 台或 200 台可部署 11 台。节点越多可靠性越高，但通信延迟也会增加，需要权衡。

### 7. `zoo.cfg` 中 `tickTime`、`initLimit`、`syncLimit` 分别表示什么？

`tickTime` 是服务端与客户端、Leader 与 Follower 的基础心跳时间；`initLimit` 是 Leader 和 Follower 初始连接时可容忍的最多心跳数；`syncLimit` 是 Leader 和 Follower 同步通信可容忍的最多心跳数。

### 8. `dataDir` 为什么不建议使用默认 `/tmp`？

`/tmp` 目录可能被系统定期清理，导致 ZooKeeper 快照、事务日志或 myid 等数据丢失。生产中应配置到稳定的数据目录，如 `/opt/module/zookeeper-3.5.7/zkData`。

### 9. `server.A=B:C:D` 各字段含义是什么？

A 是服务器编号，对应 `dataDir/myid` 文件内容；B 是服务器地址；C 是 Follower 与 Leader 交换数据的端口；D 是 Leader 故障后进行选举通信的端口。

### 10. `myid` 文件有什么作用？

ZooKeeper 启动时读取 `dataDir` 下的 `myid`，并与 `zoo.cfg` 中 `server.A` 配置匹配，以确定当前节点在集群中的身份和编号。文件内容不能有多余空格或空行。

## 三、ZNode 与监听机制

### 11. ZooKeeper 有哪些节点类型？

常见节点类型包括持久节点、持久顺序节点、临时节点、临时顺序节点。顺序节点会在路径后自动追加递增序号，临时节点会在会话结束后自动删除。

### 12. 临时节点有什么特点？

临时节点与客户端 Session 绑定，会话断开或过期后节点自动删除。它常用于服务注册、动态上下线感知、分布式锁等场景。

### 13. 顺序节点有什么特点？

顺序节点创建时会在节点名后追加单调递增的序号。如果原节点下已有节点，会在已有最大序号基础上继续递增，常用于锁排队和有序选举。

### 14. ZooKeeper Watcher 机制是什么？

客户端可以对关心的节点注册监听，当节点数据变化、节点删除或子节点变化时，ZooKeeper 会通知客户端。监听能让应用快速感知协调数据变化。

### 15. Watcher 是否永久生效？

不是。课程笔记强调 Watch 注册一次只生效一次。若希望持续监听，需要在收到通知后的回调中再次注册监听。

### 16. `ls -s` 或 `get -s` 输出中的 zxid 有什么意义？

zxid 是 ZooKeeper 事务 ID，所有修改操作都会产生唯一 zxid。zxid 越小表示事务发生越早，可用于判断事务顺序和节点状态变更先后。

## 四、一致性、选举与 ZAB

### 17. ZooKeeper 选举使用什么机制？

ZooKeeper 使用过半机制，超过半数投票才能选出 Leader 或提交事务。过半机制保证在多数节点存活时集群可以继续提供服务。

### 18. ZooKeeper 第一次启动选举规则是什么？

第一次启动时，在投票过半的前提下，服务器 ID 大的节点更容易胜出成为 Leader。

### 19. ZooKeeper 非第一次启动选举规则是什么？

优先比较 EPOCH，EPOCH 大者胜出；EPOCH 相同再比较事务 ID，事务 ID 大者胜出；事务 ID 也相同则服务器 ID 大者胜出。

### 20. ZAB 协议是什么？

ZAB 是 ZooKeeper 使用的支持崩溃恢复的原子广播协议。它借鉴 Paxos，设计为只有 Leader 处理外部写请求并发起提案，再同步给 Follower，以保证数据一致性。

### 21. ZAB 协议包含哪些基本模式？

ZAB 主要包含消息广播和崩溃恢复两种模式。消息广播用于正常写事务同步，崩溃恢复用于 Leader 故障后重新选举并保证数据一致。

### 22. ZooKeeper 的事务操作和非事务操作有什么区别？

事务操作会改变 ZooKeeper 状态，如 create、set、delete，会分配全局唯一 zxid；非事务操作如 getData 不改变状态，通常不产生新的事务日志。

## 五、生产场景与常用命令

### 23. ZooKeeper 如何实现服务器动态上下线感知？

服务端上线时在 `/servers` 下创建临时顺序节点并写入主机信息；客户端监听 `/servers` 子节点变化。服务端宕机或会话过期后临时节点消失，客户端收到通知并刷新服务器列表。

### 24. ZooKeeper 如何实现分布式锁？

客户端在锁根节点下创建临时顺序节点，序号最小者获得锁；非最小节点只监听自己前一个节点。前一个节点删除后，当前节点收到通知并尝试获取锁。释放锁时删除自己的临时顺序节点。

### 25. 原生 ZooKeeper API 实现分布式锁有哪些问题？

原生 API 会话连接是异步的，需要自行处理 CountDownLatch；Watcher 需要重复注册；递归创建删除等操作复杂；开发难度较高。Curator 对这些问题做了封装，并提供可重入分布式锁等能力。

### 26. ZooKeeper 常用客户端命令有哪些？

常用命令包括 `ls`、`ls -s`、`get`、`get -s`、`create`、`create -e`、`create -s`、`set`、`delete`、`deleteall`、`stat`、`help`。服务端常用 `zkServer.sh start|stop|status`。
