# Azkaban 高频面试题

## 整理依据

- `01_知识点.md`：多实例并发配置。
- `02_常用知识.md`：Azkaban 概论、集群部署、Flow API、依赖调度、失败重试、条件工作流、定时调度、邮件/电话告警、多 Executor 模式。

## 一、基础概念

### 1. 为什么大数据平台需要工作流调度系统？

一个完整的数据分析链路通常由 Shell、Java、MapReduce、Hive 脚本等多个任务组成，任务之间存在先后顺序和依赖关系。工作流调度系统用于统一编排、按依赖执行、定时触发、失败重试和告警，避免仅靠人工或简单 crontab 维护复杂任务链。

### 2. Azkaban 和 Oozie 的区别是什么？

Oozie 功能更全面、偏重量级，配置和使用更复杂；Azkaban 更轻量，Web 操作直观，适合对调度平台复杂功能要求不高、希望快速组织大数据任务流的场景。

### 3. Azkaban 集群模式主要由哪些组件组成？

主要包括 Web Server、Executor Server 和 MySQL。Web Server 负责项目管理、认证、计划调度和执行触发；Executor Server 负责真正执行工作流和作业；MySQL 负责存储项目、执行记录、Executor 信息等元数据。

### 4. Azkaban Web Server 和 Executor Server 的职责分别是什么？

Web Server 面向用户和调度逻辑，处理登录、项目上传、定时配置、执行触发和页面展示；Executor Server 面向任务执行，接收 Web Server 分派的 Flow，执行具体 Job，并回写执行状态和日志。

## 二、部署与配置

### 5. Azkaban 为什么需要配置 MySQL，并导入 create-all-sql？

Azkaban 依赖 MySQL 保存系统元数据，包括用户、项目、上传包、调度配置、执行历史和 Executor 信息等。导入 `create-all-sql` 用于初始化 Azkaban 所需的数据表结构。

### 6. 为什么配置 MySQL 时要调整 `max_allowed_packet`？

Azkaban 上传项目包、保存执行信息或日志时可能产生较大的数据包。如果 MySQL 的 `max_allowed_packet` 太小，可能导致连接阻塞或写入失败，因此课程中将其调整为较大的值。

### 7. 多 Executor 模式下为什么要激活 Executor？

Executor 启动后需要向系统标记为可用状态。通过访问 `executor?action=activate` 激活后，Web Server 才能将任务分派到该 Executor，否则该 Executor 即使进程启动，也不会参与调度执行。

### 8. Web Server 中 `azkaban.executorselector.filters` 有什么作用？

该配置用于选择 Executor 的过滤策略，例如 `StaticRemainingFlowSize` 关注排队任务数，`CpuStatus` 关注 CPU 占用情况。测试环境中如果启用 `MinimumFreeMemory`，可能因为内存判断导致任务不执行，课程中建议删除。

## 三、Flow 与任务执行

### 9. `azkaban.project` 文件的作用是什么？

`azkaban.project` 用于声明 Azkaban 项目的 Flow API 版本，例如 `azkaban-flow-version: 2.0`，表示使用新的 Flow API 方式解析 `.flow` 文件。

### 10. `.flow` 文件中的 `nodes`、`type`、`config` 分别表示什么？

`nodes` 表示工作流中的作业节点列表；`type` 表示作业类型，如 `command`、`javaprocess`；`config` 表示作业配置，例如要执行的命令、Java 主类、JVM 参数等。

### 11. Azkaban 如何配置作业依赖？

在子任务节点中使用 `dependsOn` 指定父任务列表。例如 JobC 依赖 JobA 和 JobB，则 JobC 的配置中写入 `dependsOn: [JobA, JobB]`，只有父任务满足条件后 JobC 才会执行。

### 12. Azkaban 如何配置自动失败重试？

可以在单个 Job 的 `config` 中配置 `retries` 和 `retry.backoff`，也可以在 Flow 全局 `config` 中配置，使所有 Job 继承重试策略。`retries: 3` 表示失败后最多重试 3 次，`retry.backoff: 10000` 表示重试间隔 10000ms。

### 13. 手动失败重试时，Parents、Ancestors、Children、Descendents 分别表示什么？

`Parents` 表示当前任务的直接上游，`Ancestors` 表示当前任务之前的所有上游，`Children` 表示当前任务的直接下游，`Descendents` 表示当前任务之后的所有下游。生产中可结合 Enable/Disable 控制失败后从指定节点或链路继续执行。

### 14. `javaprocess` 类型任务需要配置哪些核心参数？

常见参数包括 `Xms`、`Xmx`、`classpath`、`java.class` 和 `main.args`。其中 `java.class` 指定包含 `main` 方法的 Java 类，`Xms/Xmx` 控制 JVM 堆内存。

## 四、条件、定时与告警

### 15. Azkaban 条件工作流如何使用运行时参数？

父 Job 将运行结果写入 `JOB_OUTPUT_PROP_FILE` 指向的文件，子 Job 通过 `${jobName:param}` 读取父 Job 输出参数，并在 `condition` 中使用比较运算符或逻辑运算符决定是否执行。

### 16. Azkaban 预定义条件宏有哪些？

常见宏包括 `all_success`、`all_done`、`all_failed`、`one_success`、`one_failed`。默认通常是父 Job 全部成功才执行；如果配置 `one_success`，则表示父任务至少一个成功即可执行。

### 17. Azkaban 如何实现定时调度？

在 Web 页面执行工作流时选择 Schedule，并按类似 crontab 的规则配置执行时间，同时注意时区应配置为上海时区。删除定时任务可通过页面中的 remove Schedule 完成。

### 18. Azkaban 如何实现任务失败告警？

Azkaban 默认支持邮件告警，需要在 Web Server 的配置中设置 `mail.sender`、`mail.host`、`mail.user`、`mail.password` 等 SMTP 信息。若邮件不及时，可将告警邮件转发到第三方平台，再实现电话、短信等通知。

## 五、生产问题

### 19. 多 Executor 模式下任务脚本找不到怎么办？

多 Executor 模式下 Web Server 会根据策略选择某个 Executor 执行任务。若脚本只部署在部分节点，任务可能被分派到没有脚本的节点而失败。解决方案是指定特定 Executor 执行，或更推荐在所有 Executor 节点部署相同脚本和应用。

### 20. 如何允许 Azkaban 同一个 Flow 多实例并发运行？

可配置 `concurrent=true`，允许同一工作流存在多个运行实例。生产中需要确认任务是否具备幂等性、输出目录是否隔离、是否会争抢同一份资源，避免并发导致数据覆盖或重复写入。
