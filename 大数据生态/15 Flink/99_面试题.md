# Flink 高频面试题

## 整理依据

基于本目录 Flink 基础、DataStream API、时间语义、状态与容错、FlinkSQL 等课程笔记及既有面试题整理，重点覆盖 Flink 流批统一、运行时架构、算子与并行度、时间与窗口、状态管理、Checkpoint/Savepoint、ProcessFunction、Flink SQL、Connector、调优和生产问题。

## 一、基础与架构

### 1. Flink 是什么？

Flink 是一个分布式流批统一计算框架，擅长处理有状态、低延迟、高吞吐的流式计算任务。它以流处理为核心，把批处理看作有界流处理，因此可以用统一的运行时和 API 处理实时流与离线数据。

### 2. Flink 相比 SparkStreaming 的主要区别是什么？

SparkStreaming 采用微批模型，把一段时间的数据组成批次后处理；Flink 采用事件级流处理模型，数据到达后可逐条触发计算。Flink 通常延迟更低，对事件时间、状态、Checkpoint 和 Exactly-Once 语义支持更完善。

### 3. Flink 为什么说是流批统一？

Flink 将数据集抽象为流：无界流对应实时数据，有界流对应离线批数据。批处理只是有限输入上的流处理，因此 Flink 可以在同一套 DataStream API、Table API/SQL 和运行时架构上支持流批任务。

### 4. Flink 的核心组件有哪些？

核心组件包括客户端、JobManager 和 TaskManager。客户端负责提交作业；JobManager 负责作业解析、调度、Checkpoint 协调和故障恢复；TaskManager 负责执行具体 Task、管理 Slot、处理数据交换并向 JobManager 汇报状态。

### 5. JobManager 内部通常包含哪些关键角色？

JobManager 侧通常包括 Dispatcher、ResourceManager 和 JobMaster。Dispatcher 接收并启动作业；ResourceManager 负责资源申请与分配；JobMaster 负责单个作业的调度、执行状态管理、Checkpoint 协调和失败恢复。

### 6. TaskManager 和 Slot 的关系是什么？

TaskManager 是执行进程，Slot 是 TaskManager 内部划分出来的资源调度单位。一个 TaskManager 可以包含多个 Slot，作业的并行子任务会被分配到 Slot 中执行。Slot 共享使同一作业内不同算子的子任务可以共享资源，提高资源利用率。

### 7. Flink 作业从提交到执行大致经历哪些图？

用户代码首先生成 StreamGraph；经过优化和算子链合并后生成 JobGraph；调度时转换为 ExecutionGraph；最终部署到 TaskManager 上形成 Physical Graph。面试中常用这几类图解释 Flink 的作业编译、优化和运行过程。

### 8. 什么是算子链 Operator Chain？

算子链是把满足条件的相邻算子合并到同一个 Task 中执行，减少线程切换、序列化和网络传输开销。常见一对一转发且并行度相同的算子更容易被 chain 到一起；涉及 shuffle、keyBy、rebalance 等数据重分布时通常会断开算子链。

## 二、部署、资源与运行机制

### 9. Flink 常见部署模式有哪些？

常见部署模式包括本地模式、Standalone、YARN、Kubernetes 等。生产中常用 YARN 或 Kubernetes 统一管理资源，也可按 Session、Per-Job、Application 等模式提交任务。

### 10. Flink 的 Session、Per-Job 和 Application 模式有什么区别？

Session 模式先启动集群，多个作业共享集群资源，适合开发测试但作业间隔离性较弱；Per-Job 模式为每个作业单独申请集群资源，隔离性更好；Application 模式把用户 main 方法也放到集群中执行，更适合生产提交和资源隔离。

### 11. 并行度 Parallelism 如何理解？

并行度表示一个算子被拆分成多少个并行子任务执行。可以在算子级、执行环境级、提交参数或配置文件中设置。合理并行度应结合数据量、Kafka 分区数、下游吞吐、Slot 数和状态大小综合确定。

### 12. Slot 数和并行度一定相等吗？

不一定。Slot 是资源分配单位，并行度是算子子任务数量。由于 Flink 支持 Slot 共享，同一个作业中多个算子的并行子任务可能共享同一个 Slot，因此作业最大并行度通常受可用 Slot 数影响，但二者不是简单一一对应关系。

### 13. Flink 中 keyBy 的作用是什么？

keyBy 会按照指定 key 对数据进行分区，相同 key 的数据会进入同一个并行子任务，从而支持 keyed state、按 key 聚合、窗口统计等操作。keyBy 会触发数据重分布，因此可能带来网络传输和数据倾斜问题。

### 14. Flink 常见的分区策略有哪些？

常见分区策略包括 forward、shuffle、rebalance、rescale、broadcast、global、keyBy 等。forward 用于一对一传递；rebalance 轮询均衡分发；broadcast 广播到所有下游；keyBy 按 key 分区，是有状态按键计算的基础。

## 三、DataStream API 与常用算子

### 15. Flink 程序的基本开发流程是什么？

通常先创建执行环境，再读取数据源，之后进行转换处理，最后写入 Sink 并调用 execute 触发执行。DataStream 程序中的转换逻辑会被构建成作业图，提交到运行时后才真正执行。

### 16. map、flatMap、filter 的区别是什么？

map 是一进一出转换；flatMap 是一进多出或一进零出；filter 根据条件保留或丢弃元素。它们都属于基础转换算子，通常不会改变数据分区方式。

### 17. process 算子相比普通 map/flatMap 有什么优势？

ProcessFunction 能访问上下文信息、事件时间或处理时间定时器、侧输出流、状态等能力，适合实现复杂事件处理逻辑。普通 map/flatMap 更适合简单转换，无法直接使用定时器和底层时间语义能力。

### 18. Flink 的 Source 和 Sink 在生产中要关注什么？

Source 需要关注并行读取能力、offset 或消费位点管理、反压影响和故障恢复；Sink 需要关注幂等写入、事务提交、批量刷新、连接复用、下游限流以及是否支持 Exactly-Once 语义。

### 19. RichFunction 和普通 Function 有什么区别？

RichFunction 除了处理单条数据的方法外，还提供 open、close、getRuntimeContext 等生命周期和运行时上下文能力。它常用于初始化连接、加载配置、注册指标、访问状态或获取并行子任务信息。

## 四、时间语义、Watermark 与窗口

### 20. Flink 支持哪些时间语义？

Flink 支持处理时间、事件时间和摄入时间。处理时间以机器处理数据的时间为准，实现简单但结果容易受延迟影响；事件时间以数据自身携带的业务时间为准，更适合乱序数据和可重放计算；摄入时间以进入 Flink 的时间为准，使用较少。

### 21. Watermark 的作用是什么？

Watermark 用于衡量事件时间进展，告诉系统某个时间之前的数据大概率已经到齐。它主要用于处理乱序事件、触发事件时间窗口计算和清理状态。Watermark 过小会增加延迟，过大可能导致迟到数据增多。

### 22. 如何处理迟到数据？

常见方式包括设置合理的乱序等待时间、使用 allowedLateness 延迟窗口清理、通过侧输出流收集严重迟到数据，或将迟到数据写入补偿链路。具体选择取决于实时性、准确性和业务补偿要求。

### 23. Flink 常见窗口类型有哪些？

常见窗口包括滚动窗口、滑动窗口、会话窗口和全局窗口。按 key 划分后可形成 Keyed Window；不按 key 的窗口并行度通常受限。窗口可以基于处理时间或事件时间触发。

### 24. 滚动窗口、滑动窗口和会话窗口有什么区别？

滚动窗口长度固定且窗口之间不重叠；滑动窗口由窗口长度和滑动步长决定，窗口之间可以重叠；会话窗口根据数据活跃间隔动态划分，超过指定 gap 没有数据就关闭当前会话。

### 25. Window Reduce、Aggregate 和 ProcessWindowFunction 如何选择？

ReduceFunction 适合输入输出类型相同的增量聚合；AggregateFunction 更通用，可定义累加器并减少窗口缓存数据量；ProcessWindowFunction 能拿到窗口上下文和全量元素，但开销更大。生产中常组合 AggregateFunction 与 ProcessWindowFunction 兼顾性能和上下文能力。

## 五、状态、容错与一致性

### 26. Flink 中状态分为哪几类？

从作用范围看，状态分为 Keyed State 和 Operator State。Keyed State 绑定到 key，只能在 keyBy 后使用；Operator State 绑定到算子并行子任务，常用于 Source offset、分区缓存等场景。

### 27. 常见 Keyed State 类型有哪些？

常见包括 ValueState、ListState、MapState、ReducingState、AggregatingState 等。ValueState 保存单个值；ListState 保存列表；MapState 保存键值结构；ReducingState 和 AggregatingState 用于按 key 增量聚合。

### 28. State Backend 的作用是什么？

State Backend 决定状态在 TaskManager 本地如何存储以及 Checkpoint 时如何持久化。常见实现包括 HashMapStateBackend 和 EmbeddedRocksDBStateBackend。状态较小可用内存型后端，超大状态通常使用 RocksDB 降低 JVM 内存压力。

### 29. Checkpoint 的作用是什么？

Checkpoint 是 Flink 的容错机制，会周期性将算子状态和 Source 位点做一致性快照。任务失败后可以从最近一次成功 Checkpoint 恢复状态和消费位置，从而保证状态计算结果不丢失。

### 30. Flink Checkpoint 如何实现一致性快照？

Flink 基于 Chandy-Lamport 思想，通过 Checkpoint Barrier 在数据流中传播并对齐。算子收到 Barrier 后快照自身状态，并继续向下游传递 Barrier；当所有相关状态快照完成后，本次 Checkpoint 成功。

### 31. Checkpoint 和 Savepoint 有什么区别？

Checkpoint 主要用于故障自动恢复，由系统周期性触发和清理；Savepoint 通常由用户手动触发，用于版本升级、作业迁移、修改并行度或长期保留恢复点。Savepoint 更强调可管理性和兼容性。

### 32. Flink 如何实现端到端 Exactly-Once？

端到端 Exactly-Once 需要 Source 可重放、Flink 状态通过 Checkpoint 保证一致恢复、Sink 支持幂等或事务提交。对于 Kafka 等事务型 Sink，通常通过两阶段提交与 Checkpoint 协调，在 Checkpoint 成功后再提交外部事务。

## 六、Flink SQL 与生产问题

### 33. Flink SQL 中动态表是什么？

动态表是随流数据持续变化的逻辑表。Flink SQL 会把流转换成动态表，在动态表上执行持续查询，再把结果转换为追加流、撤回流或更新流输出。它是 Flink SQL 处理无界流的核心抽象。

### 34. Flink SQL 中 Append、Retract、Upsert 流有什么区别？

Append 流只追加新结果，适合无更新的查询；Retract 流会发送撤回旧值和新增新值，适合普通更新场景；Upsert 流按主键更新或插入结果，适合带唯一键的聚合结果输出到支持更新的外部系统。

### 35. Flink SQL 常见窗口 TVF 有哪些？

常见窗口 TVF 包括 TUMBLE、HOP、CUMULATE、SESSION 等。它们用于在 SQL 中基于时间列生成窗口表，再结合 GROUP BY 做窗口聚合，是新版 Flink SQL 推荐的窗口写法。

### 36. Flink SQL 中 TopN 和去重通常如何实现？

TopN 通常使用 ROW_NUMBER、RANK 等窗口函数按分区排序后过滤排名；去重常用 ROW_NUMBER 按业务主键分区，再按时间字段排序保留第一条或最后一条。生产中要注意状态大小、TTL 和输入更新语义。

### 37. Flink SQL Join 有哪些常见类型？

常见包括普通 Join、时间窗口 Join、Interval Join、Temporal Join、Lookup Join 等。普通 Join 可能长期保留状态；Temporal/Lookup Join 常用于事实流关联维表；Interval Join 适合两条流在时间范围内关联。

### 38. Flink 反压是什么？如何排查？

反压是下游处理能力不足导致上游发送受阻的现象。可通过 Flink Web UI、反压指标、Task 吞吐、busy/backpressured 状态、下游 Sink 延迟等定位瓶颈，再从扩容并行度、优化算子、批量写入、提升下游能力等方向处理。

### 39. Flink 数据倾斜如何处理？

先通过指标和 Web UI 定位倾斜算子及热点 key，再结合场景处理：过滤异常 key、提高并行度、热点 key 单独拆分、随机前缀打散后二次聚合、优化 key 设计、使用本地预聚合或调整上游分区策略。

### 40. Flink 作业常见生产调优方向有哪些？

常见方向包括合理设置并行度和 Slot、开启或优化算子链、控制状态大小和 TTL、选择合适 State Backend、调整 Checkpoint 间隔和超时、优化 RocksDB 参数、处理反压和数据倾斜、减少序列化开销、批量写 Sink、完善监控告警。
