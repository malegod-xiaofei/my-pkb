# HDFS 高频面试题

## 整理依据

基于本目录 HDFS 知识点、Shell 命令、HDFS 读写流程、NameNode/SecondaryNameNode、DataNode、数据完整性和安全模式等笔记整理，重点覆盖架构原理、读写流程、元数据、生产参数和运维命令。

## 一、基础概念与架构

### 1. HDFS 是什么？适合什么场景？

HDFS 是 Hadoop Distributed File System，是面向海量数据的分布式文件系统。它适合一次写入、多次读取的大文件离线分析场景，不适合低延迟随机读写和频繁修改的小文件场景。

### 2. HDFS 有哪些核心组件？

核心组件包括 NameNode、DataNode、SecondaryNameNode 和 Client。NameNode 管理命名空间、元数据和副本策略；DataNode 负责实际数据块存储和读写；SecondaryNameNode 负责辅助合并 FsImage 和 Edits；Client 负责文件切分、上传、下载和与 NN/DN 通信。

### 3. NameNode 主要保存哪些元数据？

NameNode 保存目录树、文件到 Block 的映射、权限、副本数、Block 列表等元数据。Block 到 DataNode 的映射通常由 DataNode 启动后通过块汇报动态构建，因此 FsImage 中不直接保存块所在 DataNode 信息。

### 4. DataNode 主要负责什么？

DataNode 负责在本地磁盘存储 Block 数据和对应元数据文件，处理客户端读写请求，定期向 NameNode 发送心跳和块汇报，并按 NameNode 指令复制、删除或恢复数据块。

### 5. HDFS 为什么不适合大量小文件？

每个文件、目录和块都会占用 NameNode 内存元数据。大量小文件会造成 NameNode 内存压力大、RPC 请求多、MapReduce 切片和调度开销高，整体吞吐下降。

## 二、块、副本与读写流程

### 6. HDFS 默认块大小为什么常见是 128M 或 256M？

块过小会导致元数据过多和寻址开销高；块过大则降低并行度。课程笔记中强调让寻址时间约占传输时间 1% 较合理，中小公司常用 128M，大规模场景常用 256M。

### 7. 1KB 文件在 HDFS 中是否占用 128M 磁盘空间？

不会。128M 是块大小上限，1KB 文件实际只占约 1KB 数据空间，但仍会占用 NameNode 元数据，因此小文件问题主要是元数据和任务调度开销，不是每个小文件都占满一个块。

### 8. HDFS 写数据流程是什么？

客户端先向 NameNode 请求创建文件，NameNode 检查文件是否存在、父目录和权限是否合法；客户端按块切分文件，请求块存放的 DataNode 列表；客户端与多个 DataNode 建立 Pipeline，以 Packet 为单位写入，DataNode 逐级转发并返回 ack；块写完后继续写下一个块，最后关闭流并通知 NameNode 完成。

### 9. HDFS 读数据流程是什么？

客户端向 NameNode 查询文件块位置；NameNode 返回块所在 DataNode 列表；客户端按就近原则选择 DataNode；DataNode 从磁盘读取数据并以 Packet 方式传输给客户端；客户端校验并写入本地目标文件。

### 10. HDFS 副本放置策略是什么？

常见副本因子为 3。如果写入端在 DataNode 上，第一个副本优先本地节点，否则随机节点；第二个副本放在不同机架；第三个副本放在第二个副本同机架的不同节点。这样兼顾写入性能、机架容灾和读取效率。

### 11. `hdfs dfs -setrep 10` 是否一定会生成 10 个真实副本？

不一定。该命令会修改 NameNode 元数据中的期望副本数，真实副本数还受 DataNode 数量和可用空间限制。例如集群只有 3 台 DataNode，最多只能形成 3 个不同节点副本。

## 三、NameNode、2NN 与元数据

### 12. FsImage 和 Edits 分别是什么？

FsImage 是某一时刻 HDFS 元数据的镜像文件；Edits 是元数据变更操作日志，采用追加写。NameNode 启动时会加载 FsImage 和 Edits 到内存，恢复最新元数据状态。

### 13. 为什么需要 SecondaryNameNode？

如果 Edits 长期追加会变得很大，NameNode 重启恢复会变慢。SecondaryNameNode 定期从 NameNode 拉取 FsImage 和 Edits，在本地合并生成新的 FsImage，再返回给 NameNode，降低 Edits 过大的风险。

### 14. SecondaryNameNode 是 NameNode 的热备吗？

不是。SecondaryNameNode 主要负责 CheckPoint 合并元数据，不负责实时接管 NameNode 服务。生产高可用应使用 HDFS HA，而不是依赖 SecondaryNameNode。

### 15. CheckPoint 触发条件有哪些？

常见触发条件包括时间周期和事务次数。默认可每 1 小时触发一次，也可按 Edits 操作次数触发，例如达到 100 万次事务后触发，并有周期检查参数。

### 16. 如何查看 FsImage 和 Edits 内容？

FsImage 可用 `hdfs oiv -p XML -i fsimage文件 -o 输出文件` 查看；Edits 可用 `hdfs oev -p XML -i edits文件 -o 输出文件` 查看。FsImage 记录目录、文件、副本和块等元数据，Edits 记录 mkdir、rename、delete 等操作日志。

## 四、DataNode、安全模式与数据完整性

### 17. DataNode 启动后做什么？

DataNode 启动后向 NameNode 注册，注册成功后周期性发送心跳和块汇报。块汇报默认间隔较长，笔记中记录为 6 小时；心跳默认约 3 秒一次。

### 18. NameNode 如何判断 DataNode 不可用？

NameNode 根据 DataNode 心跳判断节点存活。若超过超时时限未收到心跳，则认为该 DataNode 不可用。笔记中强调 `dfs.namenode.heartbeat.recheck-interval` 单位为毫秒，`dfs.heartbeat.interval` 单位为秒。

### 19. HDFS 安全模式是什么？

安全模式是 NameNode 启动或异常恢复时的保护状态。此时 NameNode 等待 DataNode 汇报块信息，当满足安全块比例后退出安全模式。安全模式下通常不能写入或删除数据。

### 20. HDFS 如何保证数据完整性？

DataNode 读取 Block 时会计算 CheckSum，并与创建时记录的校验值比较。如果发现损坏，客户端会读取其他副本，NameNode 后续会安排副本恢复。Hadoop 底层常用 CRC32 校验。

### 21. DataNode 本地一个数据块包含哪些文件？

通常包含 Block 数据文件和 `.meta` 元数据文件。数据文件保存实际内容，`.meta` 文件保存块长度、校验和、时间戳等信息。

## 五、常用命令与生产排查

### 22. HDFS 常用文件操作命令有哪些？

常用命令包括 `hdfs dfs -ls`、`-cat`、`-mkdir`、`-put`、`-get`、`-copyFromLocal`、`-copyToLocal`、`-du -h`、`-rm -r`、`-setrep`、`-count`、`-text` 等。

### 23. `hadoop fs` 和 `hdfs dfs` 有什么区别？

两者都能操作 HDFS。`hadoop fs` 是更通用的文件系统命令入口，可操作多种 Hadoop 支持的文件系统；`hdfs dfs` 更明确指向 HDFS 命令，在 HDFS 运维中更常用。

### 24. HDFS 删除文件后一定立即物理删除吗？

不一定。开启回收站时，删除数据会进入当前用户的 `.Trash` 目录，到期后再清理。若使用 `-skipTrash`，则会跳过回收站直接删除，需要谨慎使用。

### 25. 常用的 HDFS 管理命令有哪些？

`hdfs dfsadmin -report` 可查看文件系统和 DataNode 状态；`hdfs dfsadmin -safemode get|enter|leave|wait` 管理安全模式；`hdfs dfsadmin -setQuota` 可设置目录配额。
