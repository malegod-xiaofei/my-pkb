# SparkStreaming 高频面试题

## 整理依据

基于本目录 SparkStreaming Java/Scala 课程笔记及既有面试题整理，重点覆盖 DStream、微批架构、Kafka 消费方式、窗口与状态、输出、优雅关闭、背压、offset 管理和生产实时指标。

## 一、基础与架构

### 1. SparkStreaming 是什么？

SparkStreaming 是 Spark 体系中的流式处理模块，用于处理 Kafka、Flume、Socket、文件等实时数据源。它采用微批模型，把连续流数据按批次切分成一系列 RDD，再复用 Spark Core 引擎进行计算。

### 2. 什么是 DStream？

DStream 是 SparkStreaming 的基础抽象，表示连续的数据流。内部实现上，每个时间间隔的数据会被封装成一个 RDD，连续多个 RDD 组成 DStream，所以 DStream 可以理解为 RDD 在实时场景下的封装。

### 3. SparkStreaming 是真正逐条实时处理吗？

不是。SparkStreaming 是微批处理，将一段时间内的数据收集成一个批次再处理。批次间隔越小，实时性越高，但调度开销也越大；如果批处理时间超过批次间隔，容易出现任务堆积。

### 4. SparkStreaming 的主要特点有哪些？

主要特点是易用、可复用 Spark RDD 算子、容错能力较好、能与 Spark SQL/MLlib 等生态集成。其不足是端到端延迟通常高于逐条流处理框架。

### 5. 批次间隔应该如何设置？

批次间隔应大于或接近单批数据的平均处理时间，并结合延迟要求、数据量和资源情况设置。如果批次时间小于处理时间，会出现批次积压，最终导致延迟升高甚至 OOM。

### 6. SparkStreaming 为什么需要至少两个线程运行本地测试？

本地 Receiver 模式下，一个线程用于接收数据，另一个线程用于处理数据。如果只使用 `local[1]`，接收线程占用唯一线程，计算无法正常并发执行。

## 二、数据源与 Kafka

### 7. SparkStreaming 支持哪些输入源？

常见输入源包括 Kafka、Flume、Socket、文件系统、自定义 Receiver、RDD 队列等。生产中最常见的是 Kafka。

### 8. Receiver 模式消费 Kafka 的原理是什么？

Receiver 模式通过 Kafka 高阶 API 接收数据，数据先进入 Executor 内存，再由后续 Job 处理。为了保证可靠性可开启 WAL 预写日志，但会增加写分布式文件系统的开销。

### 9. Direct 模式消费 Kafka 的原理是什么？

Direct 模式由 SparkStreaming 周期性查询 Kafka 每个 TopicPartition 的 offset 范围，再按 offset 直接拉取数据生成 RDD。Kafka 分区与 RDD 分区通常一一对应，计算 Executor 主动消费，速率更可控。

### 10. Receiver 模式和 Direct 模式有什么区别？

Receiver 模式需要专门接收数据，offset 通常由 Kafka/Zookeeper 管理，可靠性依赖 WAL；Direct 模式不需要 Receiver，Spark 自己维护 offset，Kafka 分区与 RDD 分区对应，性能更高、语义更清晰。Spark 3.x 中通常只使用 Direct 模式。

### 11. Direct 模式为什么性能更好？

Direct 模式避免了 Receiver 接收后再写 WAL 的额外链路，直接基于 Kafka 副本和 offset 范围恢复数据，减少内存堆积和分布式文件系统写入开销，也简化了并行读取。

### 12. SparkStreaming 中 Kafka 分区和 RDD 分区是什么关系？

Direct 模式下 Kafka 的 TopicPartition 通常与每批 RDD 的分区一一对应，读取多个 Kafka 分区会生成相应数量的 RDD 分区，便于并行消费。

### 13. SparkStreaming 里为什么不建议随意 repartition？

repartition 会触发 Shuffle，增加批次处理时间和延迟。实时任务通常要控制链路耗时，除非分区严重不足或业务必须重新分布，否则应优先调整 Kafka 分区数或算子并行度。

## 三、转换、窗口与状态

### 14. 无状态转换和有状态转换有什么区别？

无状态转换只处理当前批次 RDD，如 map、flatMap、filter、reduceByKey；有状态转换需要跨批次保存历史状态，如 updateStateByKey、窗口操作等，通常依赖 checkpoint 或状态存储。

### 15. transform 算子的作用是什么？

transform 可以在 DStream 的每个批次上执行任意 RDD-to-RDD 操作，用于调用 DStream API 没有直接暴露的 RDD 算子，或在每个批次中动态加载维表、过滤规则等。

### 16. SparkStreaming 两个流 join 有什么要求？

两个 DStream 的批次间隔需要一致，才能在同一批次触发计算。底层本质是当前批次两个 RDD 的 join，仍然可能产生 Shuffle、数据倾斜和延迟问题。

### 17. updateStateByKey 的作用是什么？

updateStateByKey 用于跨批次维护每个 key 的状态，例如运行以来累计 WordCount、用户历史行为统计等。它需要定义状态类型和状态更新函数，并配置 checkpoint 保存状态。

### 18. 为什么 updateStateByKey 需要 checkpoint？

因为状态需要跨批次保存并在失败后恢复。checkpoint 会保存状态和部分元数据，任务重启时可从 checkpoint 恢复 StreamingContext 与历史状态。

### 19. SparkStreaming 窗口操作的原理是什么？

窗口操作是在原批次基础上再封装一层时间范围，每次计算窗口长度内的多个批次数据，并按滑动步长周期性触发。窗口长度和滑动步长必须是批次间隔的整数倍。

### 20. window 和 reduceByKeyAndWindow 有什么区别？

window 只是把多个批次合并成窗口 DStream，后续还需再执行聚合；reduceByKeyAndWindow 直接对窗口内相同 key 的 value 聚合，支持指定窗口长度、滑动步长和并行度。

### 21. reduceByKeyAndWindow 的反向 reduce 有什么优势？

带 `invFunc` 的版本会基于上一个窗口结果，加上新进入窗口的数据，减去离开窗口的数据，避免每次对整个窗口全量重算。它要求聚合函数可逆，如计数加减。

### 22. countByWindow 和 countByValueAndWindow 的区别是什么？

countByWindow 统计窗口内元素总数；countByValueAndWindow 统计窗口内每个 value 的数量，适用于窗口内按值计数的场景。

## 四、输出、容错与关闭

### 23. DStream 没有输出操作会发生什么？

如果 DStream 及其派生流没有 print、saveAsTextFiles、foreachRDD 等输出操作，计算不会被触发；如果 StreamingContext 中没有任何输出操作，整个流任务不会真正启动计算。

### 24. foreachRDD 的作用是什么？

foreachRDD 是最通用的输出操作，可以对每个批次生成的 RDD 执行任意逻辑，如写 MySQL、Redis、HDFS 或调用外部服务。它常用于生产落地。

### 25. 写数据库时连接应该在哪里创建？

不建议在 Driver 端创建连接，因为连接对象通常不可序列化；也不建议每条数据创建连接。推荐在 `foreachRDD` 内结合 `foreachPartition`，每个分区创建一次连接并批量写入。

### 26. saveAsTextFiles 为什么生产中较少直接使用？

它每个批次都会生成一批文件，容易产生大量小文件，增加 NameNode 压力和下游处理成本。生产中通常写 Kafka、数据库、Redis、HBase，或先合并/控制分区后写 HDFS。

### 27. SparkStreaming 如何优雅关闭？

可通过外部文件、数据库标记或信号控制内部程序关闭，检测到关闭条件后调用 `ssc.stop(stopSparkContext=true, stopGracefully=true)`，让当前批次处理完成后再停止，避免粗暴 kill 导致数据丢失或重复。

### 28. `spark.streaming.stopGracefullyOnShutdown` 有什么作用？

该参数用于在 JVM 关闭时触发优雅停止，让 SparkStreaming 尽量处理完当前批次再退出。生产中还常配合外部标记文件或调度系统控制停机。

### 29. 背压机制是什么？

背压机制会根据 JobScheduler 反馈的批次处理能力动态调整数据接收速率，避免上游数据过快导致内存堆积。可通过 `spark.streaming.backpressure.enabled=true` 开启。

### 30. Receiver 的静态限流和背压有什么区别？

静态限流通过 `spark.streaming.receiver.maxRate` 固定接收速率，可能导致资源利用不足；背压会根据实际处理能力动态调整速率，更适合数据量波动场景。

## 五、offset、精准一次与生产问题

### 31. SparkStreaming 如何实现 Kafka 精准一次消费？

常见思路是手动管理 offset：先读取指定 offset 范围并完成业务处理，再在业务结果成功落库后提交 offset。若结果和 offset 能写入同一事务性存储，可实现更强的一致性。

### 32. 为什么“处理完再提交 offset”仍可能重复消费？

如果业务处理成功后，在提交 offset 前程序宕机或网络异常，任务重启会从旧 offset 重新读取，导致重复消费。因此涉及金额等强一致场景，需要业务幂等或把结果与 offset 放入同一事务。

### 33. SparkStreaming 任务如何监控？

可自定义 StreamingListener 监控批次提交、处理完成、处理时间、调度延迟、输入数据量等指标；如果批次处理时间超过阈值或持续积压，应报警并排查数据量、资源和下游写入瓶颈。

### 34. 实时广告黑名单如何实现？

从 Kafka 消费广告点击数据，先过滤已在黑名单中的用户；按日期、用户、广告维度统计点击次数并写入 MySQL/Redis；若单日点击超过阈值（如 100 次）则写入黑名单，后续批次过滤。

### 35. 为什么 Kafka 到 HDFS 落盘有时不用 SparkStreaming 而用 Flume？

如果只是采集落盘，Flume 通过配置即可完成，并且对小文件、滚动策略等支持更简单；SparkStreaming 更适合需要实时计算、聚合、关联、状态处理的场景，单纯落盘会显得复杂。
