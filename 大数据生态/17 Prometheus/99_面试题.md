## 一、基础与架构

### 1. Prometheus 的核心特点有哪些？

Prometheus 是云原生监控系统，核心特点包括单二进制易部署、Pull 模型采集、内置 TSDB、基于 Label 的多维数据模型、强大的 PromQL、易与 Grafana 和 Alertmanager 集成，并支持通过 Exporter 扩展监控目标。

### 2. Prometheus 为什么采用 Pull 模型？

Pull 模型由 Prometheus Server 主动从 Exporter 或应用暴露的 `/metrics` 端点拉取数据，便于集中管理采集周期、目标健康状态和服务发现。对于短生命周期任务，则可通过 PushGateway 间接支持推送。

### 3. Prometheus 的时间序列由什么唯一确定？

一条时间序列由指标名称和一组标签唯一确定。例如 `http_request_status{code="200", path="/api"}` 与同名但标签不同的样本属于不同时间序列。标签用于过滤、聚合和多维分析。

### 4. Prometheus 生态中各组件的职责是什么？

Prometheus Server 负责采集、存储和查询；Client Libraries 用于应用埋点；Exporter 用于暴露第三方系统指标；PushGateway 用于短作业指标暂存；Alertmanager 负责告警分组、抑制、路由和通知；Grafana 负责可视化展示。

### 5. Node Exporter 的作用是什么？

Node Exporter 用于采集主机 CPU、内存、磁盘、网络等系统指标，并通过 HTTP `/metrics` 暴露给 Prometheus。Prometheus 在配置的 `targets` 中加入各节点 `9100` 端口即可采集。

### 6. PushGateway 适合什么场景？

PushGateway 适合生命周期很短、Prometheus 来不及主动拉取的批处理任务或离线作业。短作业在退出前将指标推送到 PushGateway，由 Prometheus 再从 PushGateway 拉取。

## 二、PromQL

### 7. PromQL 中瞬时向量和区间向量有什么区别？

瞬时向量返回当前时刻每条时间序列的最新样本，例如 `metric{}`；区间向量返回一段时间内的样本序列，例如 `metric{}[5m]`。区间向量常用于 `rate`、`irate`、变化趋势和告警判断。

### 8. PromQL 标签匹配有哪些方式？

常用匹配包括完全匹配 `label="value"`、不等匹配 `label!="value"`、正则匹配 `label=~"a|b"`、正则排除 `label!~"a|b"`。标签匹配用于过滤目标时间序列。

### 9. PromQL 中 `offset` 的作用是什么？

`offset` 用于时间位移查询，可以查询过去某一时间点或时间段的数据。例如 `metric offset 5m` 查询 5 分钟前的瞬时样本，`metric[1d] offset 1d` 查询昨天一天的区间样本。

### 10. PromQL 常用聚合操作有哪些？

常用聚合包括 `sum`、`avg`、`min`、`max`、`count`、`topk`、`bottomk`、`quantile` 等。可以结合 `by` 保留指定标签，或使用 `without` 去掉指定标签。

### 11. `by` 和 `without` 的区别是什么？

`by(label)` 表示聚合结果只保留列出的标签；`without(label)` 表示聚合时去掉列出的标签，保留其他标签。两者都用于控制聚合后的时间序列维度。

### 12. PromQL 中布尔运算默认行为是什么？`bool` 有什么作用？

布尔运算默认用于过滤时间序列，只有满足条件的样本会返回。加 `bool` 后返回 0 或 1，表示比较结果本身，适合需要明确真假值的场景。

## 三、部署与集成

### 13. Prometheus 配置文件中 `scrape_configs` 的作用是什么？

`scrape_configs` 用于配置采集任务，包括 `job_name` 和 `targets`。例如配置 Prometheus 自身、PushGateway、Node Exporter 或 Flink 指标端点，Prometheus 会按全局或任务级采集间隔拉取指标。

### 14. 如何判断 Prometheus 采集目标是否正常？

可以在 Prometheus Web 页面中查看 `Status -> Targets`，目标为 `up` 表示采集正常；如果为 `down`，需检查网络、端口、目标服务是否启动、路径是否正确以及防火墙配置。

### 15. Flink 如何接入 Prometheus？

课程中使用 Flink 的 Prometheus PushGateway Reporter，将 `flink-metrics-prometheus` 插件 jar 放入 Flink `lib` 目录，并在 `flink-conf.yaml` 中配置 PushGateway 的主机、端口、jobName、上报间隔等参数。

### 16. Grafana 与 Prometheus 如何集成？

在 Grafana 中添加 Prometheus 数据源，配置 Prometheus Server 地址并 Save & Test。之后可手动创建 Dashboard，也可导入社区模板 JSON，如 Flink 或 Node Exporter 模板。

### 17. 为什么 Flink 作业名最好显式指定？

如果代码中不指定 `env.execute("作业名")`，Grafana 中可能统一显示默认作业名 `Flink Streaming Job`，多个作业难以区分。生产中应指定有业务含义的作业名。

## 四、监控告警与生产问题

### 18. 如何基于 Prometheus/Grafana 判断 Flink 任务失败？

课程中利用 `flink_jobmanager_job_uptime` 持续递增的特点，若任务失败则该值不再变化。可用当前值与 `offset 30s` 的差值判断是否为 0，并据此配置 Grafana 告警。

### 19. 如何监控 Flink 任务重启？

可监控 `flink_jobmanager_job_numRestarts` 指标。若当前值减去 30 秒前的值等于 1，说明发生了一次重启，可以配置告警通知。

### 20. 网络延时或任务重启为什么会影响 `flink_jobmanager_job_uptime`？

正常情况下 uptime 会按固定频率递增；网络延时可能导致上报间隔异常，重启会导致 uptime 清零后重新上报。通过当前值与 `offset 30s` 对比，可以发现突然小于正常增量或大于 0 的异常变化。

### 21. 为什么需要第三方告警平台？

邮件告警可能接收不及时。通过 Grafana 或 Prometheus 对接第三方告警平台，可以按照分派策略和通知策略发送短信、电话、邮件等多种通知，提高生产故障响应效率。

### 22. Prometheus 监控体系中哪些指标适合作为基础告警？

基础告警通常包括目标 `up` 状态、主机 CPU/内存/磁盘/网络、应用存活、任务重启次数、任务失败、数据上报延迟等。大数据任务中还应关注作业运行时长、失败次数和关键数据链路的指标变化。
