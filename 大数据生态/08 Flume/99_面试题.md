# Flume 高频面试题

## 整理依据

基于本目录 Flume 知识点与已有面试题整理，重点覆盖 Agent 架构、Source/Channel/Sink、事务机制、常见采集场景、拓扑结构、可靠性、监控与参数调优。

## 一、基础架构

### 1. Flume 是什么？适合解决什么问题？

Flume 是一个高可用、高可靠、分布式的海量日志采集、聚合和传输系统，常用于把业务日志、服务器日志等实时采集到 HDFS、Kafka、HBase 等下游系统。

### 2. Flume Agent 由哪些核心组件组成？

一个 Agent 通常由 Source、Channel 和 Sink 组成。Source 负责采集数据，Channel 负责缓存数据，Sink 负责把数据写出到目标系统。

### 3. Flume Event 的结构是什么？

Event 是 Flume 数据传输的基本单位，由 Header 和 Body 组成。Header 保存键值对形式的元信息，Body 保存实际日志内容字节数据。

### 4. Source、Channel、Sink 分别有什么作用？

Source 接收外部数据并封装成 Event；Channel 在 Source 和 Sink 之间做缓冲；Sink 从 Channel 中取出 Event 并写入目标端，例如 HDFS、Logger、Avro、HBase 或自定义系统。

## 二、常见 Source 与 Channel

### 5. Exec Source、Spooldir Source、Taildir Source 有什么区别？

Exec Source 通常通过执行命令如 `tail -F` 读取日志，简单但进程异常时可靠性较弱。Spooldir Source 监控目录中新文件，适合采集已完成写入的文件，不能重复放入同名文件。Taildir Source 可监控多个文件并通过 positionFile 记录读取位置，支持断点续传，生产日志采集更常用。

### 6. Netcat Source 通常用于什么场景？

Netcat Source 监听指定端口并接收文本数据，常用于入门测试、调试 Flume 流程，不太适合作为生产核心日志采集方式。

### 7. Memory Channel 和 File Channel 怎么选择？

Memory Channel 基于内存，吞吐高、延迟低，但 Agent 异常退出可能丢失未消费数据。File Channel 基于磁盘，可靠性更好，但性能低于 Memory Channel。对数据可靠性要求高时优先 File Channel，对性能要求高且允许少量丢失时可考虑 Memory Channel。

### 8. Channel 的 `capacity` 和 `transactionCapacity` 分别表示什么？

`capacity` 表示 Channel 最多可缓存的 Event 数量；`transactionCapacity` 表示一次事务中可写入或读取的最大 Event 数量。通常 `transactionCapacity` 应大于等于 Source 和 Sink 的 batchSize。

## 三、Sink 与写入目标

### 9. HDFS Sink 常见滚动参数有哪些？

常见参数包括按时间滚动、按文件大小滚动、按 Event 数量滚动等。合理设置滚动参数可以避免小文件过多，也能控制数据落盘延迟。

### 10. Sink 数量是不是越多越好？

不是。增加 Sink 可以提升消费能力，但 Sink 过多会增加线程、网络和目标端压力，造成资源浪费甚至降低整体稳定性。应结合 Channel 堆积、目标端吞吐和机器资源进行调整。

### 11. Avro Sink 和 Avro Source 常用于什么场景？

Avro Sink/Source 常用于 Agent 之间的数据传输，可以构建多级采集链路，例如前端采集 Agent 汇聚到后端聚合 Agent。

## 四、事务、可靠性与重复数据

### 12. Flume 的事务机制是怎样的？

Flume 使用两个独立事务保证 Source 到 Channel、Channel 到 Sink 的传递。Source 写入 Channel 成功后提交事务；Sink 从 Channel 取出并写入目标端成功后提交事务，失败则回滚，Event 保留在 Channel 中等待重试。

### 13. Flume 会丢数据吗？

Flume 通过事务机制降低丢数据风险，但不是绝对不会丢。使用 Memory Channel 时 Agent 宕机可能丢失内存中的数据；Channel 写满后 Source 无法继续写入，外部未进入 Flume 的数据也可能丢失。高可靠场景建议使用 File Channel 并做好容量和监控。

### 14. Flume 为什么可能产生重复数据？

如果 Sink 已经把数据写到目标端，但提交事务或接收响应前发生异常，事务回滚后 Event 会再次发送，导致目标端出现重复。下游需要根据业务主键、日志唯一 ID 或幂等写入机制去重。

### 15. Spooldir Source 为什么适合采集离线落地文件？

Spooldir Source 监控目录下新增文件，文件处理完成后会标记完成，适合采集已经生成完毕、不会继续追加的文件，可靠性比直接 tail 命令更好。

### 16. Taildir Source 为什么适合生产日志采集？

Taildir Source 支持监控多个文件组，并通过 positionFile 保存每个文件的读取位置，Agent 重启后可从上次位置继续采集，适合持续追加的业务日志。

## 五、拓扑结构与路由

### 17. Replicating Channel Selector 和 Multiplexing Channel Selector 有什么区别？

Replicating 会把同一份 Event 复制到多个 Channel，适合一份数据写入多个下游。Multiplexing 会根据 Header 条件把 Event 路由到不同 Channel，适合按业务类型或日志级别分流。

### 18. Sink Processor 有哪些常见类型？

常见包括 Default Sink Processor、LoadBalancing Sink Processor 和 Failover Sink Processor。Default 对单个 Sink 生效；LoadBalancing 用于多个 Sink 负载均衡；Failover 用于主备切换，提高可用性。

### 19. Flume 常见拓扑结构有哪些？

常见拓扑包括简单串联、复制与多路复用、负载均衡与故障转移、聚合。生产中经常使用多级 Agent 汇聚日志，并通过 Failover 或 LoadBalancing 提高可靠性和吞吐。

### 20. 什么情况下使用聚合拓扑？

当多台业务机器各自产生日志，需要统一写入 HDFS 或 Kafka 时，可在每台机器部署采集 Agent，再汇聚到一个或多个聚合 Agent，集中写入下游系统。

## 六、自定义与生产调优

### 21. Flume Interceptor 的作用是什么？

Interceptor 可以在 Source 写入 Channel 前对 Event 做处理，例如补充 Header、按日志内容分类、过滤无效数据、脱敏字段等。

### 22. 什么时候需要自定义 Source 或 Sink？

当内置 Source 无法读取特定数据源，或内置 Sink 无法写入公司内部系统、特殊接口时，可以自定义 Source 或 Sink，实现特定采集和写出逻辑。

### 23. Flume 参数调优通常从哪些方面入手？

Source 侧可增加 Source 或 filegroups、调大 batchSize；Channel 侧关注类型、capacity、transactionCapacity、File Channel 多磁盘目录；Sink 侧可增加 Sink 数量、调大 batchSize，同时观察下游系统承压能力。

### 24. 如何监控 Flume 数据传输？

可以使用 Ganglia 等监控框架观察 Flume 的 Source、Channel、Sink 指标，例如 Event 接收数、发送数、Channel 堆积量、失败次数等，及时发现采集阻塞和目标端异常。

### 25. Flume Channel 堆积通常说明什么？

Channel 堆积说明 Source 写入速度大于 Sink 消费速度，可能是下游 HDFS/Kafka 慢、Sink 数量不足、batchSize 不合理、网络或磁盘瓶颈。需要结合 Sink 写出指标和目标端状态定位。
