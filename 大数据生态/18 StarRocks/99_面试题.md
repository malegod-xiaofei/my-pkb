## 一、基础与架构

### 1. StarRocks 是什么，适合哪些场景？

StarRocks 是新一代分布式 MPP OLAP 数据库，兼容 MySQL 协议，采用列式存储和向量化执行。适合 OLAP 多维分析、实时数据分析、固定报表、Ad-hoc 查询、高并发 BI 查询和统一分析场景。

### 2. StarRocks 的核心优势有哪些？

核心优势包括 MPP 并行计算、列式存储、稀疏索引、向量化执行、多副本容错、分区分桶、灵活数据模型、多种导入方式，以及可对接 Spark、Flink、Hive、ElasticSearch、MySQL 等系统。

### 3. FE、BE、Broker 的职责分别是什么？

FE 负责元数据管理、客户端连接、SQL 解析、优化、规划、调度和导入协调；BE 负责 Tablet 数据存储、计算执行、Compaction、副本管理和本地扫描；Broker 负责与 HDFS、对象存储等外部存储交互，辅助导入导出。

### 4. StarRocks 中 Tablet 是什么？

Tablet 是表经过分区和分桶后形成的逻辑分片，也是副本管理的基本单位。Tablet 会以多副本形式分布在不同 BE 上，用于实现并行查询、数据可靠性和负载均衡。

### 5. FE 高可用如何实现？

FE 通过复制协议进行选主和主从同步，元数据修改由 Leader 完成，Follower 可执行读操作。FE 节点数通常采用 `2n+1`，可容忍 `n` 个节点故障；Leader 故障后由 Follower 重新选主。

### 6. BE 在查询和导入中分别做什么？

查询时，BE 接收 FE 下发的物理执行计划，读取本地列存数据，利用索引和谓词下沉过滤，并与其他 BE 协同完成计算。导入时，BE 接收数据并写入 Tablet 的多个副本，同时向 FE 汇报结果。

## 二、表设计与数据模型

### 7. StarRocks 为什么采用列式存储？

OLAP 查询通常只访问部分列并进行聚合。列式存储能减少无关列读取，提升压缩率和缓存命中率，并配合向量化执行提升扫描和聚合性能。

### 8. Short Key Index 稀疏索引有什么作用？

Short Key Index 用于在有序数据中快速定位扫描起点。若查询条件包含排序键前缀，StarRocks 可以通过稀疏索引进行 range-scan，减少扫描行数。

### 9. StarRocks 有哪些数据模型？

主要包括明细模型 Duplicate Key、聚合模型 Aggregate Key、更新模型 Unique Key 和主键模型 Primary Key。不同模型分别面向明细存储、预聚合统计、低频更新和高频实时更新场景。

### 10. 明细模型适合什么场景？

明细模型用于保存全部历史明细数据，不做聚合或覆盖。适合日志、行为明细、宽表明细分析等场景。排序键应选择查询过滤中高频出现的维度列。

### 11. 聚合模型适合什么场景？

聚合模型适合固定汇总类查询，如访问量、消费金额、用户访问次数等。相同维度列的数据会聚合成一行，指标列按 SUM、MAX、MIN 等聚合函数合并，能够减少存储和加速聚合查询。

### 12. 更新模型和主键模型有什么区别？

更新模型支持按 Key 覆盖更新，但采用 Merge on Read，可能影响谓词下推和索引使用。主键模型通过主键约束保证同一主键仅一条记录，更适合实时、频繁更新，如 MySQL CDC 同步，但主键索引会占用更多内存。

### 13. 主键模型为什么对内存要求更高？

主键模型需要为主键建立索引，导入时会将相关主键索引加载到内存中，用于判断和处理更新冲突。因此大范围高频更新或主键过多会增加内存压力。

### 14. 排序键设计有什么原则？

排序键应优先选择过滤频率高、区分度合适、经常作为查询条件前缀的字段。查询条件命中排序键前缀时，可以显著减少扫描；如果只过滤非前缀字段，排序效果会大幅下降。

### 15. 分区分桶的作用是什么？

分区用于按时间或业务范围管理数据，便于裁剪扫描和生命周期管理；分桶将分区内数据拆成多个 Tablet，分布到不同 BE，提升并行查询能力、避免单点热点并增强可靠性。

## 三、索引与查询加速

### 16. 物化视图在 StarRocks 中有什么作用？

物化视图保存预计算结果，适合固定维度聚合或不同查询场景的加速。创建后查询可以通过优化器自动命中，通常可用 `explain` 分析是否改写到物化视图。

### 17. Bitmap 索引适合哪些列？

Bitmap 索引适合枚举型、低基数、重复值较多，并且常用于等值过滤的列。明细模型所有列可建 Bitmap 索引；聚合模型通常只有 Key 列可建。

### 18. Bloom Filter 索引适合哪些查询？

Bloom Filter 适合对高基数字段进行 `=` 或 `IN` 过滤。它能快速判断数据块中是否一定不存在某个值，从而跳过读取；但无法定位具体行，也存在误判可能。

### 19. 如何判断查询是否命中了物化视图或索引？

可通过 `EXPLAIN` 分析查询计划是否改写到物化视图；Bitmap 或 Bloom Filter 的命中情况通常可通过 Query Profile 查看扫描、过滤和索引相关信息。

### 20. Colocate Join 为什么能提升 Join 性能？

Colocate Join 要求同一 Colocation Group 中表使用一致的分桶键、副本数和副本放置方式。当 Join 列为分桶键时，同桶数据位于本地，BE 可执行本地 Join，避免网络 Shuffle。

## 四、导入与生态集成

### 21. Stream Load 适合什么场景？

Stream Load 适合从本地文件或数据流导入 CSV 等数据，通常用于 10GB 以下数据。它基于 HTTP，同步返回导入结果，可直接判断成功或失败。

### 22. Broker Load 适合什么场景？

Broker Load 适合从 HDFS、S3 等外部存储导入几十 GB 到上百 GB 的数据，支持 CSV、ORC、Parquet 等格式。它是异步导入，需要通过 SQL 查看导入任务结果。

### 23. Routine Load 的原理是什么？

Routine Load 用于从 Kafka 持续导入数据。FE 创建 Kafka 导入任务并拆分为多个 Task，Task 分配到 BE 执行；BE 通过 Stream Load 机制导入，完成后向 FE 汇报，FE 持续生成新 Task 或重试失败 Task。

### 24. Flink Connector 写入 StarRocks 的底层方式是什么？

Flink Connector StarRocks 内部会缓存并批量通过 Stream Load 写入 StarRocks，解决普通 JDBC Connector 难以满足高吞吐写入的问题，适合实时同步链路。

### 25. DataX Writer 写入 StarRocks 的流程是什么？

StarRocksWriter 通常通过 Stream Load 将 Reader 读取的数据以 CSV 或 JSON 格式批量导入 StarRocks。整体链路为 Source -> Reader -> DataX Channel -> Writer -> StarRocks。
